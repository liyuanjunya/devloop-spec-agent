# Case 5 — Meal Plan → Shopping List Auto-Sync (Mealie)

> **Spec v1** for the DevLoop Coding agent. Derived from `consolidated.md` and `selected.md`. All `code_references` are verified against `C:\Users\v-liyuanjun\Downloads\mealie\`. The companion machine-readable representation is `spec.json` (same content).

## 1. Feature summary

Add an automatic Meal Plan → Shopping List sync feature to Mealie. Every household can opt in via three new `HouseholdPreferences` fields (`auto_sync_meal_plan_to_shopping`, `auto_sync_target_shopping_list_id`, `auto_sync_run_time`) plus two infrastructure fields (`last_auto_synced_at`, `timezone`). A new scheduler task `auto_sync_meal_plan_to_shopping` registers on the existing `register_minutely` (5-min) bucket, gates per household by their configured run-time and timezone (30-min window), pulls today's `MealPlan` recipes via `RepositoryMeals.get_today(tz=...)`, filters out ingredients whose `Food.is_pantry_staple = True` (a NEW column), consolidates them via `ShoppingListService.add_recipe_ingredients_to_list`, writes to the configured (or "first active main") shopping list, and dispatches a NEW `EventTypes.meal_plan_auto_synced_to_shopping` event. A household-admin `POST /api/households/preferences/auto-sync-shopping/run-now` endpoint shares the per-household helper with the scheduler, bypasses the daily limit (but still updates `last_auto_synced_at`), and returns `AutoSyncRunResult { added_count, skipped_pantry_count, target_list_id, run_at }`. Multi-replica safety via a CAS UPDATE on `last_auto_synced_at`. Strict per-household tenant isolation enforced by the household-scoped repositories.

## 2. User stories

| ID | Priority | Story |
|---|---|---|
| US-1 | P1 | **As a** household admin, **I want to** turn on `auto_sync_meal_plan_to_shopping` and configure `auto_sync_target_shopping_list_id`, `auto_sync_run_time`, and `timezone` via `PUT /api/households/preferences`, **so that** the system knows my household opted in and when/where to sync. |
| US-2 | P1 | **As a** household member, **I want** the scheduler to run automatically at my configured `auto_sync_run_time` in my household's timezone and append today's meal-plan recipes to my chosen shopping list, **so that** my shopping list is always up-to-date without manual intervention. |
| US-3 | P1 | **As a** household admin, **I want** ingredients whose food is flagged `is_pantry_staple = True` to be excluded from the auto-sync, **so that** staples (flour, salt, oil) don't clutter my shopping list. |
| US-4 | P1 | **As a** household member, **I want** the scheduler to sync at most once per household-local day, even if the scheduler ticks multiple times within the window or runs on multiple worker replicas, **so that** my list isn't duplicated. |
| US-5 | P1 | **As a** household admin, **I want to** trigger an immediate sync via `POST /api/households/preferences/auto-sync-shopping/run-now` that bypasses the daily limit, **so that** I can re-sync after editing today's meal plan without waiting for the next scheduled run. |
| US-6 | P2 | **As a** notifier/webhook subscriber, **I want** the system to dispatch a `meal_plan_auto_synced_to_shopping` event with `{household_id, shopping_list_id, added_item_count, skipped_pantry_count}` after each successful auto-sync, **so that** downstream integrations can react. |
| US-7 | P1 | **As a** user in household A (group G1) **and a** user in household B (group G1 or G2), **I want** household A's auto-sync to **never** read household B's meal plan or write to household B's shopping list, **so that** tenant boundaries hold even when the scheduler iterates all households in a single process. |

## 5. Edge cases

| ID | Scenario | Expected behaviour |
|---|---|---|
| EC-1 | Meal plan is empty for today (household has 0 `GroupMealPlan` entries dated `today_local`). | Scheduler path: CAS UPDATE still fires (marks day as synced) to avoid repeated empty scans; `add_recipe_ingredients_to_list` is skipped; event is NOT dispatched. Manual trigger path: same; returns `AutoSyncRunResult(added_count=0, skipped_pantry_count=0, target_list_id=<resolved>, run_at=<now>)` with 200. WARNING-level log with i18n key `auto-sync.no-meal-plan-today`. |
| EC-2 | `auto_sync_target_shopping_list_id` is None AND household has 0 shopping lists. | Both paths return early with `target_list_id=None, added_count=0, skipped_pantry_count=0`. Scheduler logs WARNING with `auto-sync.no-target-list` and does NOT update `last_auto_synced_at` (so the household will retry on the next valid tick). Manual trigger returns 200 with same shape. Event is NOT dispatched. |
| EC-3 | `auto_sync_target_shopping_list_id` points to a list that was deleted between configuration time and scheduler tick. | With `ondelete="SET NULL"` on the FK, the column is auto-nulled by the DB on the cascade delete. Behaviour then matches EC-2 fallback (or "first active list" if one still exists). The scheduler does NOT attempt to write to the deleted id and does NOT 500. |
| EC-4 | Recipe `R` is referenced by a meal-plan entry but `R` was deleted (or its `recipe_id` no longer resolves) between meal-plan creation and sync run. | `mp.recipe` is None (via `selectinload`). The iteration filter `if not mp.recipe_id or not mp.recipe: continue` skips this entry silently. `added_count` and `skipped_pantry_count` do not include it. No exception. |
| EC-5 | Household timezone changes from `America/New_York` to `Asia/Tokyo` mid-day. | `now_local` is computed fresh on each tick from `ZoneInfo(prefs.timezone)`. The `today_start_utc` derived from the new tz may move the daily boundary, possibly enabling a second sync in the same wall-clock day. This is accepted (admin opted in to the change); `last_auto_synced_at` CAS still prevents double-runs within the **new** household-local day. |
| EC-6 | Food is flagged `is_pantry_staple = True` **after** today's sync has already added that food to the list. | No retroactive removal. The previously-added item remains in the shopping list with all its `recipe_references`. The flag only affects **future** sync runs. Documented in the i18n message and route response that pantry-staple flagging is forward-looking only. |
| EC-7 | `last_auto_synced_at` exists in the future (e.g., server clock drifted backward 5 minutes; the column was set 5 minutes ago in what was then "now"). | CAS predicate `(last_auto_synced_at IS NULL OR last_auto_synced_at < :today_start_utc)` only compares against today's local-midnight UTC, not "now". A future `last_auto_synced_at` (within today) still skips the run, which is correct. A `last_auto_synced_at` from a previous local day (regardless of clock drift) allows the run. |

| SCN-3 | `last_auto_synced_at` storage is a single column on `HouseholdPreferencesModel`. This couples the CAS race condition to the entire row's lifecycle — if another preference update lands at the same instant, SQLAlchemy may end up with a stale write (the PUT route's update would overwrite our CAS marker). To prevent this, the CAS UPDATE uses raw `session.execute(text(...))` (bypassing the ORM unit-of-work for this column) and commits immediately. The PUT route should NOT touch `last_auto_synced_at` via the regular update path — add an explicit `exclude={"last_auto_synced_at"}` in the model_config or in the route's pre-write transform to ensure it's never written by `repos.household_preferences.update(self.household_id, new_pref)`. | Confirm in implementation: `last_auto_synced_at` is server-only; the `UpdateHouseholdPreferences` schema either omits it from PUT processing or the route filters it out before `update()`. |
