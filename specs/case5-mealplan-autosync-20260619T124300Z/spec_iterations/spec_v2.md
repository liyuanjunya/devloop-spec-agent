## 1. Feature summary

Add an automatic Meal Plan → Shopping List sync feature to Mealie. Every household can opt in via three new user-configurable `HouseholdPreferences` fields (`auto_sync_meal_plan_to_shopping`, `auto_sync_target_shopping_list_id`, `auto_sync_run_time`) plus one new user-configurable `timezone` field and one **server-only** `last_auto_synced_at` infrastructure column. A new scheduler task `auto_sync_meal_plan_to_shopping` registers on the existing `register_minutely` (5-min) bucket — Mealie has no native 30-min bucket and input §"实现约束" forbids creating one — and **gates per-household** to a 30-min window so the *effective per-household execution* is once per household-local day. The task fetches today's `MealPlan` entries via `RepositoryMeals.get_today(tz=...)`, re-fetches the full `Recipe` per entry (since `ReadPlanEntry.recipe` is `RecipeSummary` and has no `recipe_ingredient`), filters out ingredients whose `Food.is_pantry_staple = True` (a NEW column on `IngredientFoodModel`), and feeds the filtered ingredients into the existing consolidation seam — `ShoppingListService.add_recipe_ingredients_to_list` (which internally invokes `get_shopping_list_items_from_recipe` → `bulk_create_items` → `can_merge` + `merge_items`). The target list defaults to the **oldest household shopping list** (`created_at` asc) when `auto_sync_target_shopping_list_id` is None. A new `EventTypes.meal_plan_auto_synced_to_shopping` event is dispatched. A household-admin `POST /api/households/preferences/auto-sync-shopping/run-now` endpoint shares the per-household helper with the scheduler, bypasses the daily limit (still updating `last_auto_synced_at` per the matrix in §6), and returns `AutoSyncRunResult { added_count, skipped_pantry_count, target_list_id, run_at }`. Multi-replica safety via a single CAS UPDATE on `last_auto_synced_at`. Strict per-household tenant isolation enforced by the household-scoped repositories.

## 3. User stories

| ID | Priority | Story |
|---|---|---|
| US-1 | P1 | **As a** household admin, **I want to** turn on `auto_sync_meal_plan_to_shopping` and configure `auto_sync_target_shopping_list_id`, `auto_sync_run_time`, and `timezone` via `PUT /api/households/preferences`, **so that** the system knows my household opted in and when/where to sync. |
| US-2 | P1 | **As a** household member, **I want** the scheduler to run automatically at my configured `auto_sync_run_time` in my household's timezone and append today's meal-plan recipes to my chosen shopping list, **so that** my shopping list is always up-to-date without manual intervention. |
| US-3 | P1 | **As a** household admin, **I want** ingredients whose food is flagged `is_pantry_staple = True` to be excluded from the auto-sync, **so that** staples (flour, salt, oil) don't clutter my shopping list. |
| US-4 | P1 | **As a** household member, **I want** the scheduler to sync at most once per household-local day, even if the scheduler ticks multiple times within the 30-min window or runs on multiple worker replicas, **so that** my list isn't duplicated. |
| US-5 | P1 | **As a** household admin, **I want to** trigger an immediate sync via `POST /api/households/preferences/auto-sync-shopping/run-now` that bypasses the daily limit, **so that** I can re-sync after editing today's meal plan without waiting for the next scheduled run. |
| US-6 | P2 | **As a** notifier/webhook subscriber, **I want** the system to dispatch a `meal_plan_auto_synced_to_shopping` event with `{household_id, shopping_list_id, added_item_count, skipped_pantry_count}` after each successful auto-sync, **so that** downstream integrations can react. |
| US-7 | P1 | **As a** user in household A (group G1) **and a** user in household B (group G1 or G2), **I want** household A's auto-sync to **never** read household B's meal plan or write to household B's shopping list, **so that** tenant boundaries hold even when the scheduler iterates all households in a single process. |

## 6. Manual trigger: `last_auto_synced_at` update matrix

This table resolves cons-006 / NC-1 by enumerating every manual-trigger outcome (`bypass_daily_limit=True`) and stating whether `last_auto_synced_at` is updated. The scheduler path (`bypass_daily_limit=False`) follows FR-20 step 6 (CAS) for the success path and never reaches steps 7-8 if CAS skips.

| Manual outcome | FR-20 step where return occurs | `last_auto_synced_at` updated? | Returned shape |
|---|---|---|---|
| Auto-sync disabled (`auto_sync_meal_plan_to_shopping = False`) | step 2, but `bypass_enabled_check=True` skips this — proceeds | n/a | proceeds to next step |
| Window check | step 3, but `bypass_daily_limit=True` skips this — proceeds | n/a | proceeds |
| No target list resolvable (EC-2) | step 4 | **No** (no useful work done) | 200 `{added=0, skipped=0, target=None, run_at=now}` |
| No meal plans today (EC-1) | step 7 manual branch | **Yes** | 200 `{added=0, skipped=0, target=<resolved>, run_at=now}` |
| Successful sync (≥1 ingredient written or skipped) | step 8 manual branch | **Yes** | 200 `{added=N, skipped=M, target=<resolved>, run_at=now}` |
| Exception during sync work (e.g., DB error) | step 8 catch (route layer) | **No** (rolled back) | 500 — surfaced by FastAPI default handler |
| Configured target list deleted between config and call (EC-3) | step 4 (since FK `SET NULL` makes resolution fall through to FR-22) | **No** if EC-2 path; **Yes** if EC-1/sync path | matches EC-2 or EC-1/sync outcome |

## 7. Edge cases

| ID | Scenario | Expected behaviour |
|---|---|---|
| EC-1 | Meal plan is empty for today (household has 0 `GroupMealPlan` entries dated `today_local`). | FR-20 step 7. **Scheduler path**: CAS UPDATE already fired at step 6 (marks day as synced to avoid repeated empty scans); `add_recipe_ingredients_to_list` is NOT called; event NOT dispatched. **Manual path**: writes `last_auto_synced_at` per §6 matrix; returns `AutoSyncRunResult(added_count=0, skipped_pantry_count=0, target_list_id=<resolved>, run_at=now_utc)`. WARNING-level log with i18n key `auto-sync.no-meal-plan-today`. |
| EC-2 | `auto_sync_target_shopping_list_id is None` AND household has 0 shopping lists. | FR-20 step 4. Both paths return early with `target_list_id=None, added_count=0, skipped_pantry_count=0`. Scheduler logs WARNING with `auto-sync.no-target-list` and does NOT update `last_auto_synced_at` (so the household retries on the next valid tick — admin may create a list later). Manual returns 200 with the same shape and also does NOT update `last_auto_synced_at`. Event NOT dispatched. |
| EC-3 | `auto_sync_target_shopping_list_id` points to a list that was deleted between configuration time and scheduler tick. | With `ondelete="SET NULL"` on the FK (FR-1), the column is auto-nulled by the DB on the cascade delete. Behaviour then matches EC-2 fallback (or the oldest-list rule if one still exists). The scheduler does NOT attempt to write to the deleted id and does NOT 500. |
| EC-4 | Recipe `R` is referenced by a meal-plan entry but `R` was deleted (or its `recipe_id` no longer resolves) between meal-plan creation and sync run. | FR-17: `full_recipe = group_recipes_repo.get_one(mp.recipe_id, "id")` returns None → entry is skipped silently. `added_count` and `skipped_pantry_count` do not include it. No exception. |
| EC-5 | Household timezone changes from `America/New_York` to `Asia/Tokyo` mid-day. | `now_local` is computed fresh on each tick from `ZoneInfo(prefs.timezone)`. The `today_start_utc` derived from the new tz may move the daily boundary, possibly enabling a second sync in the same wall-clock day. Accepted (admin opted in to the change); the CAS guard still prevents double-runs within the **new** household-local day. |
| EC-6 | Food is flagged `is_pantry_staple = True` **after** today's sync has already added that food to the list. | No retroactive removal. The previously-added item remains in the shopping list with all its `recipe_references`. The flag only affects **future** sync runs (the next day's `add_recipe_ingredients_to_list` will skip it). No i18n / runtime message — this semantic is documented in release notes only (no FR carries a runtime help string for this). |
| EC-7 | `last_auto_synced_at` exists in the future (e.g., server clock drifted backward 5 minutes; the column was set 5 minutes ago in what was then "now"). | CAS predicate `(last_auto_synced_at IS NULL OR last_auto_synced_at < :today_start)` only compares against today's local-midnight UTC, not "now". A future `last_auto_synced_at` (within today) still skips the run, which is correct. A `last_auto_synced_at` from a previous local day (regardless of clock drift) allows the run. |
| EC-8 | A meal-plan recipe contains a sub-recipe (`RecipeIngredient.referenced_recipe is not None`), and one of the sub-recipe's ingredients has `is_pantry_staple = True`. | **Intentional limitation**: FR-17's pantry filter only inspects top-level `recipe_ingredient` entries on the meal-plan recipe. The sub-recipe ingredients are pulled in by `get_shopping_list_items_from_recipe` recursion (`shopping_lists.py:344-355`) **after** the filter ran, so pantry staples inside sub-recipes are not skipped. Documented as NC-4; deferred scope. The non-pantry sub-recipe ingredients still get the same `(food_id, unit_id)` consolidation. |

## 10. Constraints (carried + amended from input.md §"实现约束")

- MUST reuse `mealie/services/scheduler/` existing abstractions (no parallel scheduler). Concretely: register via `SchedulerRegistry.register_minutely` (the only 5-min bucket Mealie has); per-household window gating inside the task achieves the 30-min effective interval.

- Multi-replica deployment: use DB row-level lock (`SELECT ... FOR UPDATE SKIP LOCKED`) OR `last_auto_synced_at` CAS to guarantee one worker per household per day. **Selected: CAS** (SQLite-compatible — `FOR UPDATE SKIP LOCKED` is Postgres-only).
- Timezone: MUST use household-configured timezone (fallback **directly to** server default UTC; **no group default** exists in Mealie); **forbidden** to use `datetime.now()` without `tz=` argument.
- All code must follow Mealie repo conventions per `.github/copilot-instructions.md`:

  - Dependency injection via `Depends(get_repositories)`, `Depends(get_current_user)`.
  - Type hints mandatory; mypy-compatible.
  - Run `task dev:generate` after Pydantic schema changes (regenerates TS types + `__init__.py` exports).
  - Only `en-US` locale modified for i18n.
  - Migrations tested against SQLite + PostgreSQL.
  - Use `uv run ...` for Python commands.
