# Architecture Review (v1)

## Verdict: NEEDS_REFINE

The spec's central architectural decision — reuse `UserToRecipe.is_favorite` rather than introduce a new `user_favorite_recipe` table — is correct, well-evidenced, and aligned with Mealie's 3-layer convention, `AllRepositories` factory, `BaseUserController`, and the per-group recipe scoping pattern. However, three architectural gaps need resolution before implementation: (1) the spec assumes `/api/recipes` is the anonymous read path, but anonymous reads actually live on a separate `PublicRecipesController`; (2) FR-008's reference to `column_aliases` as the projection extension point is mechanically wrong — that property only feeds order-by and queryFilter, never the SELECT clause; (3) the user-deletion cleanup path is missing while the recipe-deletion path has it, creating an asymmetric cascade story that US-6 oversimplifies.

## Critical issues

(none — the storage-reuse decision, repository placement, and auth model are all architecturally sound)

## High issues

- **ARCH-H-001**: US-4 AC3 / FR-007 / FR-009 cannot be satisfied by touching only `mealie/routes/recipe/recipe_crud_routes.py`. The anonymous recipe read path is a separate controller.
  - Location: US-4 AC3 ("unauthenticated request to supported public recipe reads"); FR-007 code_references; FR-009 code_references; self-concern #4.
  - Evidence: `mealie/routes/recipe/recipe_crud_routes.py:85` mounts the recipe router on `UserAPIRouter(prefix="/recipes", route_class=MealieCrudRoute)`. `UserAPIRouter.__init__` (`mealie/routes/_base/routers.py:23-24`) injects `Depends(get_current_user)` for every route. `RecipeController` (`recipe_crud_routes.py:89`) extends `BaseRecipeController` → `BaseCrudController` → `BaseUserController`, and `BaseUserController` (`mealie/routes/_base/base_controllers.py:139`) also depends on `get_current_user`. So `/api/recipes` and `/api/recipes/{slug}` reject anonymous requests with 401; they cannot produce `favorited=false` for an anonymous reader. The actual anonymous path is `PublicRecipesController` in `mealie/routes/explore/controller_public_recipes.py:20-125` (extends `BasePublicHouseholdExploreController`, no auth dependency), mounted at `/api/explore/...`. Neither FR-007 nor FR-009 reference this controller or `mealie/schema/recipe/recipe.py` projection through the public path.
  - Suggested action: Either (a) scope US-4 AC3 to the `/api/explore/groups/{slug}/recipes` + `/recipes/{recipe_slug}` endpoints and add an FR that explicitly enumerates the public controller as a touch point, or (b) drop US-4 AC3 and document anonymous behavior as out of scope. Self-concern #4 should be re-rated `confirmed_problem`, not "needs implementation-time verification".

- **ARCH-H-002**: FR-008's pointer to `RepositoryRecipes.column_aliases` as the projection extension point is mechanically incorrect — that property is only consumed for ORDER BY and queryFilter, never for SELECT projection.
  - Location: FR-008 code_references (`mealie/repos/repository_recipes.py:36-52, 72-93`) and FR-009 code_references (`mealie/repos/repository_recipes.py:40-52`).
  - Evidence: `column_aliases` is read in exactly two places: `mealie/repos/repository_generic.py:370` (passed into `QueryFilterBuilder.filter_query` for queryFilter expressions like `favoritedBy.id = "..."`) and `:414` (`order_attr = self.column_aliases.get(order_attr.key, order_attr)` inside `add_order_attr_to_query`). The page_all SELECT at `:342-344` materializes rows via `eff_schema.loader_options()` + `model_validate`, never touching `column_aliases`. Adding a `favorited`/`favorite_count` entry to `column_aliases` would make those names sortable/filterable but would NOT cause the values to appear in the JSON response. An implementer following FR-008 verbatim would be misled.
  - Suggested action: Reword FR-008 to drop the `column_aliases`/`_get_rating_col_alias` analogy and instead specify the actual projection mechanism: either (a) add fields to `RecipeSummary` plus a SQLAlchemy `column_property`/`hybrid_property` on `RecipeModel`, or (b) add fields to `RecipeSummary` and hydrate them in a post-query batched lookup keyed by page item ids (the pattern most consistent with FR-009's "one batched lookup for all page item ids"). The existing `_get_rating_col_alias` (`repository_recipes.py:72-93`) is a precedent for *sorting* by user-specific rating — it is not a precedent for *returning* user-specific fields in the recipe payload, which is what US-4 needs.

- **ARCH-H-003**: US-6 AC2/AC3 oversimplifies the cleanup story. Recipe deletion already cleans up favorites in application code; user deletion does not, and there is no FK cascade.
  - Location: US-6 AC2, US-6 AC3, FR-001 self-concern #2.
  - Evidence: The 2024-03-18 migration (`mealie/alembic/versions/2024-03-18-02.28.15_d7c6efd2de42_…py:164-171`) creates `users_to_recipes` FKs with no `ondelete` clause (default is `NO ACTION`/`RESTRICT`). Recipe deletion is handled in app code: `mealie/repos/repository_recipes.py:110-128` (`_delete_recipe`) explicitly issues `sa.delete(UserToRecipe).where(UserToRecipe.recipe_id == recipe.id)` before deleting the recipe row, with the comment "first remove UserToRecipe entries so we don't run into stale data errors". `RepositoryUsers.delete` (`mealie/repos/repository_users.py:55-65`) has no equivalent — it calls `super().delete()` and removes the user's filesystem directory but never touches `users_to_recipes`. On Postgres this would raise an IntegrityError on user deletion if any UserToRecipe row references the user; on SQLite (default FK enforcement off in some setups) it could leak orphans that still affect `favorite_count`.
  - Suggested action: Split US-6 into two ACs with different remedies. For recipe deletion: cite `_delete_recipe` as the proof of correctness and require a regression test against that path. For user deletion: explicitly require either (a) extending `RepositoryUsers.delete` to delete UserToRecipe rows for the user (mirroring `_delete_recipe`), or (b) adding an `ondelete="CASCADE"` migration on the `user_id` FK. The current "add the smallest migration needed" phrasing is too vague for the implementer to choose between the two paths.

## Medium issues

- **ARCH-M-001**: `UserToRecipe` event listener fires on every favorite toggle and forces a recipe rating recomputation, which SC-004 does not measure.
  - Location: SC-004 (only measures `GET /api/recipes` p95); FR-003/FR-004 (idempotent POST/DELETE on `/favorites`).
  - Evidence: `mealie/db/models/users/user_to_recipe.py:46-53` registers `after_insert`, `after_update`, and `after_delete` listeners on `UserToRecipe`, all of which call `update_recipe_rating` (`:36-43`), which loads the related `RecipeModel` and sets `recipe.rating = -1` to flag a recompute. Every `POST /favorites/{slug}` and `DELETE /favorites/{slug}` therefore performs an extra SELECT + UPDATE on `recipes`. If a future hydration strategy materializes a denormalized `favorite_count` column on `recipes` (one valid reading of FR-009), it would interact with this listener and need to participate in the same after-write recompute pipeline. Spec is silent on this interaction.
  - Suggested action: Either add a non-regression check to SC-004 covering favorite POST/DELETE latency, or document explicitly in FR-009 that any denormalized `favorite_count` column must hook into (or coexist cleanly with) the existing `update_recipe_rating_on_insert_or_delete` listener.

- **ARCH-M-002**: FR-003 idempotency claim is racy under concurrent POSTs.
  - Location: FR-003 ("create a `UserToRecipe` link if absent, otherwise update the existing row…without duplicate rows"); SC-001.
  - Evidence: The existing implementation pattern at `mealie/routes/users/ratings.py:54-76` (`set_rating`) reads `get_by_user_and_recipe` and then conditionally calls `create` or `update`. Two concurrent POSTs from the same user against the same recipe slug could both see `user_rating=None` and both attempt `create`. The `UniqueConstraint("user_id", "recipe_id", name="user_id_recipe_id_rating_key")` at `mealie/db/models/users/user_to_recipe.py:19` prevents the duplicate row but causes the second `create` to raise `IntegrityError` → 500. This is pre-existing behavior, not introduced by this spec, but FR-003's "no duplicate rows" is currently only enforced by raising 500 on the loser of the race. SC-001's "second POST returns success" assertion would pass for sequential requests and fail for concurrent ones.
  - Suggested action: Either (a) accept the pre-existing behavior and tighten SC-001 to say "sequential", or (b) require an UPSERT (Postgres `ON CONFLICT DO NOTHING/UPDATE`, SQLite `INSERT OR IGNORE` followed by UPDATE) so the second POST also returns 200. The existing migration at `:62-76` already uses both UPSERT idioms, so the patterns are available.

- **ARCH-M-003**: FR-006's "explicit compatibility decision" burden is overstated; the only live consumer of `/api/users/self/favorites` is a single parametrized test.
  - Location: FR-006, US-3 AC4, self-concern #1.
  - Evidence: Server route at `mealie/routes/users/crud.py:38-40` returns `UserRatings[UserRatingSummary]`. Frontend TS client `frontend/app/lib/api/user/users.ts:62-64` defines `getSelfFavorites()`, but it calls `routes.ratingsSelf` (= `/api/users/self/ratings`, line 27), **not** `/users/self/favorites`. The only place that hits `/users/self/favorites` is `tests/integration_tests/user_recipe_tests/test_recipe_ratings.py:64` (the `use_self_route=True` branch), via `tests/utils/api_routes/__init__.py:192` (`users_self_favorites = "/api/users/self/favorites"`). So a breaking response-model change has exactly one test caller to update, plus a stale TS route constant `usersSelfFavoritesId` (`users.ts:39`) that is defined but never invoked.
  - Suggested action: Reword FR-006 / US-3 AC4 to specify the actual blast radius (one test, one stale TS route definition) so the implementer can confidently choose "break and update test" rather than the heavier "alias + dual contract" path.

## Self-concerns verdicts (the writer's `self_concerns` list)

- **"US-3/FR-006 — Current `/api/users/self/favorites` returns rating summaries, while the input asks for paginated recipes"** — verdict: **resolved (and cheaper than the spec implies)**
  - Explanation: Confirmed via `mealie/routes/users/crud.py:38-40`. Cross-checked the live caller set: only `tests/integration_tests/user_recipe_tests/test_recipe_ratings.py:64` calls this endpoint at runtime; frontend `users.ts:62-64` `getSelfFavorites` actually routes to `/self/ratings`. The spec's compatibility wording is safe but overcautious — see ARCH-M-003.

- **"US-6/FR-001 — Migration/FK cascade behavior for users_to_recipes may not clean rows on all supported databases"** — verdict: **confirmed_problem**
  - Explanation: Verified at `mealie/alembic/versions/2024-03-18-02.28.15_d7c6efd2de42_…py:164-171` — `ForeignKeyConstraint` is created without `ondelete`. Application-level cleanup exists for recipe deletion (`mealie/repos/repository_recipes.py:110-128`) but NOT for user deletion (`mealie/repos/repository_users.py:55-65` does not touch `users_to_recipes`). See ARCH-H-003 for the recommended split.

- **"US-4/FR-008 — favorite_count visibility semantics are not explicit in current code"** — verdict: **uncertain**
  - Explanation: The `RecipeModel.favorited_by` relationship (`mealie/db/models/recipe/recipe.py:68-74`) has no visibility filter — a raw count via `UserToRecipe.recipe_id == X AND is_favorite == True` is a global count across all users who favorited, regardless of household, group, or recipe `settings.public`. The spec's stated assumption ("counts should follow recipe endpoint visibility/group scoping") is a *product* decision, not a *code-evidenced* convention. Spec correctly flags this as an open product question; no architectural objection.

- **"US-4/FR-007 — Unauthenticated recipe route handling may need optional-auth wiring not fully inspected here"** — verdict: **confirmed_problem (impact larger than the self-concern suggests)**
  - Explanation: Per ARCH-H-001, the impact is not "optional-auth wiring on the existing route" — `/api/recipes` is hard-gated by `UserAPIRouter` (`mealie/routes/_base/routers.py:23-24`) AND `BaseUserController` (`base_controllers.py:139`) and cannot be made optional-auth without a redesign. Anonymous reads live on `PublicRecipesController` (`mealie/routes/explore/controller_public_recipes.py:20-125`), a separate file with its own routes, its own controller base, and its own `cross_household_recipes` repo. Self-concern should be promoted from "uncertain" to "implementation must extend the public explore controller as well."

## Summary

The spec's macro-level architecture is right: reuse `UserToRecipe.is_favorite`, extend `RepositoryUserRatings`, add self routes through `BaseUserController` / `UserAPIRouter`, and respect the `group_recipes` scoping pattern. Storage reuse is correct given the 2024 migration that already consolidated favorites into `users_to_recipes`, and the `AllRepositories` factory is not bypassed. The needs-refine verdict comes from three concrete gaps: (1) US-4 AC3's anonymous-read requirement targets a controller (`recipe_crud_routes.py`) that is not anonymous-accessible — the real public path is `controller_public_recipes.py`; (2) FR-008's `column_aliases` reference is mechanically wrong as a projection extension point; (3) US-6 conflates recipe deletion (already handled in app code) with user deletion (not handled, with no FK cascade). Fixing these three plus the medium-severity items above will produce a spec the implementation team can execute without architectural surprises.
