## 5. Existing service-disabled / 503 pattern

**None for 503 specifically.** Verified pattern survey:

- `mealie/routes/users/registration.py:24-28` returns **403** for "User Registration is Disabled" — semantically the closest existing precedent for "feature globally disabled", but uses the wrong status (403 implies auth/authorization; 503 is the correct "service unavailable" semantic the spec calls for).
- `mealie/routes/recipe/recipe_crud_routes.py:320-325` returns **400** for "OpenAI image services are not enabled" — the existing image endpoint's gate, also wrong status per the new spec.
- `mealie/routes/handlers.py:1-31` only registers a 422 debug handler for `RequestValidationError`; no global 503 mapping.

| Field | Recommendation |
|-------|----------------|
| 503 idiom | `raise HTTPException(status_code=status.HTTP_503_SERVICE_UNAVAILABLE, detail=ErrorResponse.respond(self.t("recipe.image.feature-disabled")))`. `status` is already imported in `recipe_crud_routes.py:19`. Mirror this for `feature-disabled`, `openai-failed`, `parse-failed`, etc. |
| Gate composition | Inline at the top of the route handler — check `self.settings.OPENAI_ENABLE_IMAGE_RECIPE` first (env-var kill switch), then `self.group.ai_provider_settings.image_provider_enabled` (per-group setting); either falsy → 503. Same i18n key for both, since from the client's perspective the feature is unavailable either way. |

## Cross-perspective questions

1. **Data**: We propose calling `RecipeService.create_one(recipe)` rather than `repos.recipes.create(recipe)` directly so the household-settings injection (`services/recipe/recipe_service.py:208-218`), user-rating creation (225-233), and timeline event (236-244) all happen. Please confirm that's the right level — the existing `OpenAIRecipeService` returns the `Recipe` schema but does **not** itself call `create_one`; the wrapping `RecipeService.create_from_images` does (line 351). We'll keep that split.

2. **Data**: Should we add a new TimelineEventType / event subject like `recipe.recipe-created-from-image` (currently we'd reuse the generic `RECIPE_CREATED_EVENT_SUBJECT` from `services/recipe/recipe_service.py:40`)? If yes, the corresponding i18n key and TimelineEventType enum member belong to the data side.

3. **Data**: For the rate-limit counter — if we go with the DB-backed variant (per §4 row "限流"), we'd need a new model in `mealie/db/models/` and an alembic migration. The data perspective should advise: is a brand-new table the cleanest, or should we piggyback on an existing audit/log table? (No obvious candidate exists today.) MVP recommendation here is in-memory; flagging for the data side because if a DB version is ever needed, it's their domain.

4. **Data**: `OpenAIRecipe.recipe_yield` is `str | None` (`schema/openai/recipe.py:56-59`), while `Recipe.recipe_yield_quantity` is `float` (`schema/recipe/recipe.py:128`). The current mapper only sets `recipe_yield` (`recipe_service.py:607`), leaving `recipe_yield_quantity=0`. Is that intentional, or should the LLM be asked to also produce a numeric quantity? If yes, we'd need to extend the `OpenAIRecipe` schema, which is a data-side change.

5. **Data**: We're proposing to **remove** the `data_service.write_image(f.read(), "webp")` call at `services/recipe/recipe_service.py:354-355` for the new flow. The existing test `tests/integration_tests/user_recipe_tests/test_recipe_create_from_image.py:72-75` asserts that the resulting recipe **has** an image (GETs `original.webp`). Spec §4 隐私 says we must NOT persist. **Please confirm the test must be updated** to assert the recipe has **no** image (only fields the LLM produced), and that this behavior change is acceptable from the data/UX perspective.

6. **Data**: The new orchestrator must respect both gates (env `OPENAI_ENABLE_IMAGE_RECIPE` AND per-group `image_provider_enabled`). What happens if a request reaches the orchestrator with env=true but per-group=false? Should that be 503 (no service-config) or 412 (precondition-failed because the group hasn't configured AI)? We're going with **503 + same `recipe.image.feature-disabled` key** for both for simplicity, but data side may have a stronger opinion since they own the per-group settings UX.
