## User stories

| ID | Pri | Story | Acceptance gate (cross-ref to AC) |
|---|---|---|---|
| US-1 | P1 | As an **authenticated household user**, I upload a single image file (`multipart/form-data` field `image`) to `POST /api/recipes/create/image` and receive an HTTP 201 with a fully-formed `Recipe` JSON body containing the LLM-extracted `name`, `recipe_ingredient[]`, and `recipe_instructions[]`, persisted under my household. | AC-01, AC-02 |
| US-2 | P1 | As a **server administrator**, when I leave `OPENAI_ENABLE_IMAGE_RECIPE` unset (default `false`) any **authenticated** call to the endpoint returns HTTP 503 with i18n key `recipe.image.feature-disabled`. **Unauthenticated** calls return HTTP 401 (FastAPI auth dependency runs first); per the precedence chain in FR-25. | AC-03, AC-04, AC-04b |
| US-3 | P1 | As a **defensive user**, with the feature enabled I cannot send a 6 MB JPEG (rejected 413 + `recipe.image.too-large`), nor a `text/plain` file labeled `image/jpeg` (rejected 415 + `recipe.image.unsupported-mime`), nor a `.svg` claiming `image/png` (rejected 415); the rejection happens before the OpenAI call is made and before any temp file is left on disk. | AC-05, AC-06, AC-07 |
| US-4 | P1 | As an **authenticated user**, my 11th successful request within a rolling 60-minute window returns HTTP 429 + `recipe.image.rate-limited`; counts reset as the oldest entry ages out of the window. Only requests that pass earlier checks (auth, gate, size, MIME) are recorded; rejected attempts do not consume quota. | AC-08 |
| US-5 | P1 | As a **security reviewer**, when OpenAI returns malformed JSON, raises a network/timeout error, or violates the `OpenAIRecipe` pydantic schema, the HTTP response is exactly `{"detail": {"message": "recipe.image.openai-failed"}}` or `… "recipe.image.parse-failed"` at status 422 — **never** containing raw LLM text, model name, stack trace, original exception message, or upstream error body. Corresponding server logs (at any log level including DEBUG) contain only the i18n key, the user id, and the response length, never the response content; `raise … from None` is used to suppress the exception cause chain so neither HTTP nor logs can leak via the chained traceback. | AC-09, AC-10, AC-11, AC-11b |
| US-6 | P2 | As an **authenticated user**, after the endpoint succeeds I can immediately call existing read/update routes (`GET /api/recipes/{slug}`, `PUT /api/recipes/{slug}`, `PUT /api/recipes/{slug}/image` for adding a cover later) on the returned recipe — the new endpoint emits the same `EventTypes.recipe_created` event-bus message as `_finish_recipe_from_web`, so consumers (webhook, timeline) react identically. | AC-12 |

## self_concerns

- **SCN-1**: `python-magic` requires `libmagic1` system library; Mealie's official `docker/Dockerfile` does not install it. We sidestep by adopting `filetype` (pure-python) per NC-002. Risk if CR insists on `python-magic`: small Docker-image change + native build burden on macOS dev machines without `brew install libmagic`.
- **SCN-2**: Rate-limit storage is per-process. To keep the per-user/hour contract honest, FR-11 hard-disables the feature when `settings.WORKERS > 1`; default is 1 worker per `settings.py:432`. Multi-worker deployments must wait for the DB-backed migration deferred per NC-003. Risk: operators who set `UVICORN_WORKERS=2` won't notice the feature is off until they read the ERROR log line at startup.
- **SCN-3**: Prompt-injection mitigation per FR-17 is textual and **scope-limited**: its goal is to prevent image text from changing model role/tool behavior. It does **not** guarantee that recipe output is free of malicious-looking content. Layered defenses: `cleaner.clean(recipe_data, self.translator)` HTML/script-strips the converted recipe before persistence; front-end renders user-recipe text in safe contexts (existing behavior). No model-side guarantee available.
- **SCN-4** (from history R4): Re-introducing a 60s timeout reverses PR #6227 (`96acc6fc` 2025-09-23 "fix: Remove explicit timeout from OpenAI image API Call"). Some legitimate large-image vision calls historically exceeded 60s. Spec text mandates ≤60s — risk is documented in EC-06 and is non-negotiable. Mitigation: log timeouts at WARN so operators can correlate.
- **SCN-5**: Removing `data_service.write_image(f.read(), "webp")` (`recipe_service.py:354-355`) orphans the UI button shipped in PR #5647 ("Select recipe cover image when creating recipe from multiple images"). PR description must call this out; front-end fix is out-of-scope per `intent.scope.out_of_scope`.
- **SCN-6** (resolves ARCH-H-2/CONS-C-004): The previous spec proposed `logger.warning(..., exc_info=True)` on the failure path. v2 explicitly drops `exc_info` because the orchestrator uses `raise … from None`, which suppresses both `__cause__` and `__context__`. Without `from None`, the chained traceback could include the leaky `str(e)` from `openai.py:309`. Trade-off: server-side debugging of OpenAI failures loses the upstream traceback; mitigated by the warn line still containing the i18n key and user id, and the timing/timeout cases being unambiguous from the i18n key.
- **SCN-7** (resolves ARCH-H-1/EXEC-CITE-3): FR-23 mutates `mealie/schema/openai/_base.py:33-35` globally to redact raw response bodies from DEBUG logs. This is a **global change** that also affects the URL-scrape and other OpenAI parse paths. Net effect is positive (parity privacy hardening across all OpenAI callers); risk is loss of raw-response debugging on those non-image flows when DEBUG logs are sampled. Mitigation: the log line still includes `cls.__name__` and response length; full debug requires reproducing locally with a print added.
