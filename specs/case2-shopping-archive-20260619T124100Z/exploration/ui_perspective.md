## Cross-perspective questions

1. **(For History/spec perspectives)** The current `ShoppingListSummary` and `ShoppingListOut` differ — `Summary` has `recipe_references: list[...]` (non-optional, see schema line 219) while `Out` has `list[...] = []` (defaulted, line 252). When we add `archived_at` and `archived_by`, should both classes have **identical defaults** to avoid generated-type drift? Recommend: both `Optional[datetime] = None` and `Optional[UserSummary] = None`. **Confirm with backend perspective.**

2. **(For History/spec perspectives)** Spec says default `GET` *queries* don't return `archived_at` (backward compat). Does this also apply to `GET /api/households/shopping/lists/{id}` for an active list? If yes, then the field is conditional on the query string, which **complicates codegen** (the TS type can't model "field present only when query param X is set"). Strongly recommend: **always return `archived_at` (with `null` for active)**; only the *filtering behavior* of the collection endpoint is governed by `?archived=`. → Critical decision for UI implementation simplicity.

3. **(For UI perspective itself, sanity-check)** Should the archive action live in `[id].vue` (detail page only) or also on the `index.vue` list cards (a "swipe to archive" or context menu on each card)? The existing detail page is the natural place (you need to be confident you're archiving the right list). But the index also has a delete button per card (line 113–121). Recommend: **detail page only for archive**, since archive requires "all items checked" precondition that is easier to verify when viewing the list. The index could surface "Restore" (unarchive) since archived lists are read-only and don't need precondition checks.

4. **(For History/spec perspectives)** The spec mandates `409` for "frozen" mutations, but the existing FastAPI controllers use `status.HTTP_400_BAD_REQUEST` for analogous "object-state-doesn't-allow-this" errors (e.g., the label-settings validation at controller line 239). **Confirm 409 is the right code** — it semantically fits ("conflict with current state") but is inconsistent with the existing 400 precedent in this very file. **Recommend 409** to match HTTP semantics, but call out the inconsistency in the design doc.

5. **(For UI perspective itself)** The detail page currently auto-polls every 5 seconds. If user A is viewing an archived list (read-only banner shown) and user B unarchives it, the polling response will flip `isArchived` to `false` mid-session. Should the banner just disappear and the edit affordances reappear? Or should we lock the page state at load time and require an explicit refresh? Recommend the former (live update) for consistency with all other Mealie polling behaviors, but with a brief toast: `shopping-list.list-was-unarchived`.

6. **(For History/spec perspectives)** The `item_count` and `total_estimated_amount` fields in the archive event payload — `item_count` is trivially `len(shopping_list.list_items)`, but **`total_estimated_amount` is not currently a tracked concept** in the schema (no `price` / `cost` / `amount` columns on `ShoppingListItem`). Spec says "if available" — confirm we just omit it for now and add it later when a "price tracking" feature lands. Otherwise this single bullet expands scope dramatically.

7. **(For UI perspective itself)** Where should the "View Archived Lists" affordance live? Three options:<br>
   (a) A toggle on `/shopping-lists` like the existing "Show All" checkbox — minimal nav change, mixes archive UI into the active page.<br>
   (b) A new sidebar entry "Archived Shopping Lists" → `/shopping-lists?archived=true` (or `/shopping-lists/archived`) — most discoverable, adds sidebar noise.<br>
   (c) A button in the page header that swaps query params — discoverable only from the page itself.<br>
   **Recommend (a)** — matches existing UX pattern (the page already has `?disableRedirect=true`), avoids sidebar bloat, and the "Show All" checkbox sits right where the new toggle naturally belongs. **UI implementation perspective should confirm.**

8. **(For UI perspective itself)** Confirm the deep-link behavior: if a user has bookmarked `/shopping-lists/{archived-list-id}`, opening it should still work (show the list in read-only mode with the archived banner). This implies the detail endpoint **must** return archived lists by default. Cross-checks with cross-perspective Q3 in History — they should be answered together.
