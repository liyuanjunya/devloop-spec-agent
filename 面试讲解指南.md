## 1. 一分钟自述（电梯演讲）

> 给个**新场景**：面试官问"介绍一下你这个项目"，你只有 60 秒。

DevLoop Spec Agent 是一个**多智能体（multi-agent）系统**，专门解决 **AI 写软件需求文档（spec）时不靠谱**这个真实问题。

输入是一句话需求（例如「给商品页加用户评论功能」）和一个**真实的 git 代码仓库**；输出是一份**带行号引用、可被下游 code agent 直接消费**的工程级 spec（同时输出 `spec.md` 给人看 + `spec.json` 给机器用）。

跟「让 ChatGPT 写一段需求」的本质差别有三点：
1. **9 个阶段的流水线**：意图理解 → 代码探索 → 方案选型 → 写作 → 校验 → 评审 → 元评审 → 重写 → 落盘，每个阶段都有 pydantic schema 强约束，写错就抛错重试。
2. **跨家族评审硬约束**：写作用 Claude，评审用 GPT，**配错直接启动报错**——避免「同家族 LLM 互相说好话」这个 2024–2025 学术界已经反复证实的偏见。
3. **19 道防御机制**：每一道都对应一个我在 6 个真实 Mealie 项目 case 上亲眼见过的失败模式，包括行号引用造假、软语言（"如有需要"）、md/json 漂移、回归性重写（越改越烂）等，**18/19 道在生产 case 里真的被触发过**。

技术栈：Python 3.11+，pydantic v2，async-first，11,678 行生产代码，17,194 行测试，**537 个测试全过**，覆盖率 79.84%（核心防御 95–100%）。

## 2. 它到底解决什么真问题？为什么不能"让 GPT 直接写就完了"

### 2.1 表面问题

> 普通人会想：让 LLM 写一份需求文档不就完了吗，能有多难？

**实测**：用 Claude Opus 4.7 单次直接出 spec，放到 6 个 Mealie 真实功能（购物清单归档、收藏夹、菜单 N+1 优化、跨域定时同步、LLM 图片识别等）上，**6/6 都至少有 1 个评审维度判 NEEDS_REFINE**，平均每份 spec：
- 有 **≥4 条 file:line 引用是错的**（指向不存在的行、不存在的函数）
- 有 **≥1 处 markdown 版 spec 和 JSON 版 spec 内容对不上**（人看的写"归档"，机器读的写"删除"）
- 有 **≥1 个软语言**（"或类似实现"、"如有需要"、"TBD"）让下游 code agent 卡死

### 2.2 真痛点（从下游视角看）

| 痛点 | 不解决会怎样 |
|---|---|
| 行号引用造假 | code agent 跳去看，找不到，只能瞎猜或放弃 |
| 软语言 | "把价格做成可配置或类似方案" — code agent 不知道做哪种，多半选错 |
| md/json 漂移 | PR review 的人看 md，自动化测试读 json，结果两边对不上 |
| 漏掉一类用户 | 加了「评论」功能没考虑「删除评论」，上线后才发现 |
| 越改越烂 | 第一版有 5 个问题，让 LLM 改完变成 7 个问题 |
| 同厂模型互捧 | Claude 写完让 Claude 评，分高得离谱，跨厂复核全栽 |
| 隐藏式选项 | spec 里写"option A or B or C, TBD"，没人决策就跑去 code，浪费 1–2 天 |

这些问题**单个看都很小**，但叠在一起意味着「自动化软件开发 agent」根本跑不起来——下一棒（plan/code/test agent）一旦基于烂 spec 工作，错误是**指数放大**的。所以 spec 阶段的质量是**整条流水线的瓶颈**。

### 2.3 为什么"prompt 工程"解决不了

直觉上很多人会问：在 prompt 里加一句"不要写 TBD、不要瞎引用行号"不就行了？

**实测过，不行**：早期版本就是 prompt 提示型的，实测**修正率 60–75%**——LLM 经常忽略 prompt 里的禁令。所以本系统把这些约束**从"提示"升级为"硬约束"**——pydantic schema 层直接抛 `ValidationError`，强制写作器重写。改成硬约束之后**修正率 ≈ 100%**（你结构上就写不出来不合规的 spec）。这是本项目的核心工程哲学之一：**block，不要 hint**。

## 3. 整体架构：9 个阶段 + 4 类 agent + 5 个机械校验器

### 3.1 鸟瞰图（一张图看懂）

```
┌──────────────────────────────────────────────────────────────┐
│  用户输入: "给商品页加用户评论" + repo 路径                  │
└──────────────────────────────────────────────────────────────┘
                        ↓
  Stage 0 · Preflight        (纯 Python，规则拒绝太短/太空的输入)
                        ↓
  Stage 1 · Repo Skeleton    (tree-sitter 扫码 → 1024 token 摘要 + 缓存)
                        ↓
  Stage 2 · 意图理解         (Claude analyzer → GPT skeptic → Claude verifier)
                        ↓                                ⤴ 3 轮"假设-质疑-收敛"
  Stage 3 · 5 视角探索       (data / api / ui / test / history 并行 · Claude)
                        ↓     ↑ B2 自动检测覆盖盲区 → 定向重探 ≤3 次
  Stage 3.5 · 巩固整合       (consolidator · Claude)
                        ↓
  Stage 4 · 3 方案脑暴       (conservative / balanced / aggressive · Claude)
                        ↓     ↑ GPT 评估 → Claude 选最优
  Stage 5 · 写作器           (writer · Claude Opus 4.7)
                        ↓     ↑ 自反思字段 self_concerns
  Stage 5.5 · 机械校验       (5 个纯 Python validator · 毫秒级)
                        ↓     citation / trace matrix / md-json roundtrip / pytest collect / escalation
  Stage 6 · 4 + 1 视角评审   (architecture / completeness / executability / consistency · GPT)
                        ↓     ↑ C1 红队 (条件触发，安全敏感时启用)
  Stage 6.5 · 元评审 (B4)    (Claude · 去重 + 跨视角冲突识别)
                        ↓
            ┌──────── ALL PASS? ────────┐
            ↓ NO                        ↓ YES
  Stage 7 · 重写器               Stage 9 · 落盘
            ↓                    spec.md / spec.json / review.json / trace.jsonl
  Stage 7.5 · A1 回归守卫
            ↓
       回到 Stage 5.5 重新校验
       (最多 20 轮 · 连续 3 轮无进展或回归预算耗尽 → halt + needs_review)
```

### 3.2 三个关键概念，先记住

| 概念 | 一句话解释 |
|---|---|
| **不变式（invariant）** | 系统启动时就强制满足的硬约束，违反则崩溃。例如 writer 和 reviewer 必须不同厂。 |
| **机械校验 vs LLM 评审** | 机械 = 纯 Python，0 幻觉，100% 可复现；LLM 评审 = 灵活但有概率漏看。两者**互补**。 |
| **收敛底线（convergence floor）** | 系统知道什么时候**该停下来问人**，而不是继续假装 fix。 |

### 3.3 模型分工（为什么选这两个）

| 角色 | 模型 | 厂 | 为什么 |
|---|---|---|---|
| writer / rewriter / 自反思 / explorer / consolidator / plan generator / plan selector / verifier / analyzer / meta-reviewer | **Claude Opus 4.7** | Anthropic | 长上下文、强代码理解、写作流畅 |
| skeptic / plan evaluator / 4 个评审 / adversarial 红队 | **GPT-5.5** | OpenAI | 跨厂复核，避开同家族偏见 |

**为什么 meta-reviewer 又用回 Claude？**——这是有意设计：meta 不是再评 spec，而是**识别 4 个 GPT 评审之间的冲突**。同家族 meta 反而对 GPT 侧盲区更敏感。

## 4. 跟着一个真实例子走一遍：case2 「购物清单归档」

> 让面试官跟你走完一整条流水线，比看 100 张架构图都有用。
> 完整 artifacts 在 `specs/case2-shopping-archive-live-new-20260620T120351Z/`。

### Step 0 · 输入

```bash
devloop spec "给购物清单加归档功能，已归档的不出现在主列表，但可以恢复" \
  --repo C:\Users\v-liyuanjun\Downloads\mealie
```

### Step 1 · Preflight + Repo Skeleton（< 1 秒，无 LLM）

* 输入长度 / 关键词 / 安全字符检查 → 通过
* tree-sitter 扫 mealie 整个 repo → 提取 ~1024 token 的代码摘要（按 commit hash 缓存到 SQLite，下次同 commit 命中即返回，节省 ~30s + ~$0.02）

### Step 2 · 意图理解（Claude × 3 轮）

* `analyzer` 提出 **3 个假设**：
  - H1: 归档 = 软删除（status 字段）
  - H2: 归档 = 移到 archive 表
  - H3: 归档 = 单纯 UI 隐藏
* `skeptic`（用 GPT，跨家族）逐条挑刺：
  - "H1 没解释为何需要『恢复』按钮"
  - "H2 增加新表，迁移成本高，与现有 multitenant 模式冲突"
  - "H3 不解决数据查询需求"
* `verifier`（Claude）综合：选 H1（带 archive_at 时间戳），写入 `ConfirmedIntent.primary`，scope = `["data", "api", "ui", "test"]`

> 这一步的核心**反直觉**点：**不是让 LLM 一次想清楚**，而是**逼它先提多个假设、再被另一家模型质疑、最后让自己收敛**。多假设 + skeptic 是抗"LLM 第一反应往往片面"的对策。

### Step 3 · 5 视角并行探索（Claude × 5 个 explorer）

每个 explorer 有 **12 个代码工具**（code_search / file_read / references / navigation / git_tools 等），独立调查自己的角度：

* **data explorer**：找到 `ShoppingList` 模型在 `mealie/db/models/...`，发现已有 `created_at`/`updated_at` 字段，没有 archive 相关字段
* **api explorer**：找到 `/api/groups/shopping/lists` 路由，存在 CRUD 但无 archive endpoint
* **ui explorer**：找到 `frontend/composables/use-shopping-lists.ts`，发现列表组件
* **test explorer**：找到 `tests/unit_tests/test_routes/test_shopping_lists/`，发现现有 fixture
* **history explorer**：用 `git log` 找类似改造（recipe favorites 加过类似字段，可参考）

每个 explorer 输出 `Perspective` 对象，含 `code_references`（指向具体行号）+ `findings`（自然语言总结）+ `gaps`（自己也不确定的地方）。

**接下来 B2 覆盖盲区检测**（纯 Python）扫一遍 5 份 Perspective：
- **singleton 检测**：只有 1 个 explorer 提到的关键事实 → 可能是漏看
- **conflict 检测**：两个 explorer 说法不一致
- **sparse 检测**：某个 scope 只有 1 个 finding，密度太低
本 case 检测结果：无 gap → 直接进入 consolidator。

> case3（GAP-B2-live）实测：检测出 2 个 gap，触发 2 次定向重探（targeted re-explore），最多并行 3 个补丁式 explorer 修补，cap 在 3 避免发散。

### Step 4 · 3 方案脑暴 + 评估 + 选型

3 个 plan_generator 并行生成：
* **conservative**：在 `ShoppingList` 加 `archived_at: datetime?` 字段，所有现有查询加 `WHERE archived_at IS NULL`
* **balanced**：方案 A + 单独的 `/archive` `/unarchive` endpoint + 前端 filter
* **aggressive**：引入 `ArchiveService` 抽象，未来支持 recipe / meal plan / category 等任意实体归档

`plan_evaluator`（GPT，跨家族评估）按 5 维评分：可行性 / 完整性 / 与现状一致性 / 风险 / 性价比。
`plan_selector`（Claude）综合评估结果选 balanced——"覆盖核心需求，不过度抽象"。

### Step 5 · 写作器输出 v1 spec

`writer`（Claude Opus 4.7）拿到 `ConsolidatedExploration` + `SelectedApproach`，输出一份完整的 pydantic `Spec` 对象，包含：
* `summary` / `intent_summary` / `user_stories`（3 条）
* `functional_requirements`（8 条 FR，**每条带 `code_references` 指向具体文件:行号**）
* `success_criteria`（5 条 SC，含可量化阈值如 "p95 < 100ms"）
* `entities`（数据模型变更）
* `edge_cases`（如"归档已删除的列表如何处理"）
* `self_concerns`（writer 自己也不确定的地方，例如 "是否需要批量归档 endpoint，未在原始需求中提及"）

**重点**：`self_concerns` 是设计要的——逼 writer **承认自己不确定的东西**，下游 reviewer 可以专门关注这些点。

### Step 5.5 · 5 个机械校验器并行跑（纯 Python，毫秒级）

| 校验器 | 检查什么 | 结果 |
|---|---|---|
| **A5 citation_verifier** | 每条 `code_references` 的 file 真实存在、line range 在文件长度内、symbol 真实在引用行段 | 通过 |
| **B1 md_json_roundtrip** | spec.json → 渲染成 .md → 解析回 → 与原 .json 字节级一致 | 通过 |
| **B3 trace_matrix** | 每条 FR 都有对应的 SC + user story；反之亦然，无孤儿 | 通过 |
| **C2 test_executability** | spec 提到的所有 test stub 用 `pytest --collect-only` 跑一遍，看能否被发现 | 通过 |
| **F3 escalation** | 任何 `Concern.evidence_gap` 含 "or A or B or C" 的多选项必须升级到 `Spec.needs_clarification` | 通过 |

**任何一项失败 → 立即注入 HIGH executability issue 进 review，让下一轮 review 强制看到。**

> 这一层是项目的**护城河**：它不是"reviewer 可能看出来"，是"结构上写错就崩"。

### Step 6 · 4 视角并行评审（GPT × 4，C1 红队条件触发）

4 个 reviewer 并行，每个用**不同的 mental model** 评同一份 spec：
* **architecture**：方案是否与现有 multitenant + group 模式吻合？是否引入新的耦合？
* **completeness**：3 条 user story 是否覆盖了"恢复"、"批量归档"、"已归档列表的搜索"等隐含场景？
* **executability**：下游 code agent 拿到这份 spec 能直接动手吗？哪里仍有歧义？
* **consistency**：FR-3 说"归档后保留 30 天"，SC-2 说"无保留期限"——内部冲突！

每个 reviewer 输出独立的 `Review`，含若干 `ReviewIssue`（severity = critical/high/medium/low + axis + 引用位置）。

**C1 adversarial 红队**：本 case 因为 intent 不含 security/auth/upload/llm 等敏感词，**不触发**；case6（LLM 图片识别）则触发，并独立找到 5 个其他 reviewer 全漏的 CVE 级 bug。

### Step 6.5 · 元评审 B4（Claude）

把 4 份 review 合并：
* **去重**：4 个 reviewer 都提了"FR-3 vs SC-2 冲突" → 合并为 1 条
* **跨视角冲突识别**：architecture 说"应该用 archive 表"，executability 说"加字段就够"——meta 标记 `cross_axis_conflicts` 让 writer 知道需要二选一
* **优先级排序**：按 severity + 影响面排序，输出 `MetaReviewResult.prioritized_actions`

### Step 7 · 重写器 + Step 7.5 · A1 回归守卫

`rewriter`（Claude）拿到 v1 spec + 合并后的 review + meta 优先级，输出 v2 spec。

**A1 回归守卫**（纯 Python）对比 v1 和 v2 的 critical+high 总数：
* v1: 3 个 critical + 5 个 high = **8**
* v2: 0 个 critical + 0 个 high = **0** ✅

**all_pass = True** → 进入 Stage 9 落盘，结束。**本 case 1 轮收敛**。

> 如果 v2 比 v1 还差（critical+high 增加），A1 自动 **revert 回 v1** 并附上"你这次改回归了，请保留 v1 的优点"的反馈，重新让 rewriter 写。回归预算最多 2 次，耗尽则 halt 并标 `needs_review`。

### Step 9 · 落盘

```

├── spec.md                      # 最终给人看的 markdown
├── spec.json                    # 同样内容给机器消费
├── review.json                  # 4 reviewer 的合并 verdict
├── trace.jsonl                  # 每次 LLM 调用 + 工具调用的全量 telemetry
├── intent/                      # Stage 2 的 3 轮中间产物
│   ├── initial_hypotheses.json
│   ├── skeptic_round_1.json
│   └── confirmed.json
├── exploration/                 # 5 个视角的探索产物 + 合并
│   ├── data_perspective.json
│   ├── api_perspective.json
│   ├── ui_perspective.json
│   ├── test_perspective.json
│   ├── history_perspective.json
│   └── consolidated.json
├── approach/                    # Stage 4 的 3 候选 + 评估 + 选型
│   ├── candidate_conservative.json
│   ├── candidate_balanced.json
│   ├── candidate_aggressive.json
│   ├── evaluation.json
│   └── selected.json
└── spec_iterations/             # 每一轮 spec + 每个 reviewer 的反馈
    ├── spec_v1.md / spec_v1.json
    ├── review_v1_architecture.json
    ├── review_v1_completeness.json
    ├── review_v1_executability.json
    └── review_v1_consistency.json
```

**所有中间产物全留档** → 任何决策都可被审计、复现、回放。

## 6. 五大核心创新（面试官会眼前一亮的）

> 每一点都要能讲清楚 **"问题是什么 → 你为什么这么做 → 别的做法为什么不行"**，否则容易被追问翻车。

### 创新 1 · 跨家族评审是硬约束，不是配置

**问题**：2024–2025 年多篇论文（Self-Preference in LLM Judges、Verbosity Bias、Cohort Bias）证明：**同家族 LLM 评同家族 LLM 输出，分数虚高 5–18%**。在 writer→reviewer→rewriter 闭环里这种偏置**复利放大**——每一轮重写都被"友善"评审通过，最终 spec 在跨家族复核时大面积 fail。

**你的做法**：在 `devloop/llm/routing.py::ModelRouter.__init__` 的构造函数里，启动期就 `raise ValueError` 检查 `primary_provider != cross_review_provider`。这意味着配置错了**程序根本启动不起来**，**生产中无法绕过**。

**别的做法为什么不行**：
* 写在文档里"建议跨家族"——会被新成员/紧急 hotfix 忽略
* 写成 warning log——日志噪音里没人看
* 写成 config 项让人勾选——默认值的选择决定 99% 用户的行为
**只有 fail-fast 才能保证不变式被 100% 遵守。**

### 创新 2 · 机械校验完全替代 LLM 自评的 4 类失败

**问题**：LLM 自评有概率漏看，且很难量化（你怎么知道这次它会漏？）。

**你的做法**：识别出 4 类**结构性**问题，全部用纯 Python 校验器替代 LLM 评审：

| 防御 | 替代了 LLM 的什么自评 | 结果 |
|---|---|---|
| **A4 软语言** | 让 LLM 自己检查"我有没有写 TBD" | pydantic `field_validator` + 9 词黑名单 + Unicode confusables (62K 行同形字表) + 零宽字符识别 → **结构上写不出来** |
| **A5 行号引用** | 让 reviewer 验证 file:line 真实存在 | 直接打开文件比对 → 1000 条引用 / 0.9 秒，**LLM 永远做不到这个吞吐** |
| **B1 md/json 漂移** | 让 reviewer 对比 .md 和 .json | spec.md 由 .json 机械生成，roundtrip 不一致直接 `AssertionError` → **物理上不可能漂移** |
| **B3 trace matrix** | 让 reviewer 检查 FR ↔ SC ↔ test 覆盖 | 矩阵算法 200×200 / 23ms，0 漏 |

**别的做法为什么不行**：
* "再加一个 LLM reviewer 专门看这个"——成本翻倍，且仍然有概率漏
* "在 writer prompt 里加禁令"——实测修正率 60–75%
* **机械校验 = 0 概率 + 0 token + 毫秒级**，没有理由不做

### 创新 3 · 收敛底线（convergence floor）

**问题**：朴素的 review-rewrite 循环假设"多迭代几次总能收敛到 0 issue"。但 **case5（跨域定时同步：cron + multitenant + i18n + event bus）实测**：跑 5 轮后**卡在 5 个 critical+high 不再下降**。

剩下的不是 LLM 的写作问题，而是**产品决策问题**：
* "transactional outbox vs internal commits 怎么选" — 涉及一致性 vs 延迟权衡
* "当天用户没有 meal plan 时，sync 走哪条路径" — 产品语义问题

**这些问题 LLM 不询问 PM 是答不出来的**。如果让循环继续跑，rewriter 就会反复给同一个问题换名字、伪装成"修复"。

**你的做法**：
1. **A1 回归守卫**：连续 3 轮 critical+high 数不下降（stagnant）即 halt
2. 标记 `spec.metadata.needs_review = True`（区别于 preflight 的 hard fail）
3. **完整 trace + 当前最佳 spec 仍然落盘**，便于人工介入

**别的做法为什么不行**：
* "把 max_iteration 调到 100" — 钱烧光了还在原地打转
* "让 LLM 自己判断是否需要人工" — LLM 倾向于"再试一次"，永远不会主动叫停
**好的系统知道什么时候该停下问人，比"伪装收敛"诚实得多。** 这一点在和评审者讨论时常常被低估。

### 创新 4 · 红队第五维度（C1 adversarial）

**问题**：4 个常规 reviewer（architecture / completeness / executability / consistency）问的都是"spec 自身是否良好"——但**安全攻击者问的是另一个问题**："如果 code agent 严格按 spec 实现，能怎么打进去？"

**你的做法**：当 intent.scope 含 `security` / `auth` / `external_integration` / `payment`，或 primary 含 `upload` / `image` / `llm` / `password` / `token` 等关键词时，**自动加入第 5 个 reviewer**——`adversarial` 角色，用攻击者 mental model 扫一遍。

**真实战绩（case6 LLM 图片识别菜谱）**：C1 独立发现 **5 个其他 4 reviewer 全漏的 CVE 级 bug**：
1. **CRITICAL** 存储型 XSS（CVSS ~7.5）— OpenAI Vision 转写含 `<img onerror=...>` 的菜谱页，新 `create_one` 路径未调 sanitizer，存库后任何查看者触发
2. **HIGH** EXIF prompt-injection 绕过 — `SYSTEM: ignore prior...` 写进 JPEG `UserComment`，绕过 Layer-1/2 text guard
3. **HIGH** 8192×8192 JPEG → 256 Vision tiles → 单次请求成本 64×，配合 rate limit 10/h 单用户放大 ~640×
4. **HIGH** Rate-limit DoS-on-self — attacker 用 1×1 黑像素故意触发 OpenAI parse 失败 10 次，烧光合法用户的 hourly slot
5. **MEDIUM** httpx DEBUG logger 泄露 base64 原图 — FR-019 承诺 "no LLM at any level" 但只控制了 Mealie 自己的 logger

**别的做法为什么不行**：
* "在 architecture reviewer prompt 里加上『考虑安全』"——会被淹没在其他维度里，不专心
* "对所有 spec 都跑 adversarial"——非安全 case 浪费 token，且 false positive 升高
**条件触发 + 独立 mental model = 高 ROI**。

### 创新 5 · 回归感知重写（A1 + revert）

**问题**：纯粹"迭代到收敛"在 LLM 重写里**不收敛**——重写常常**修一个、坏俩个**。case6 v2 复现过：架构 axis 从 1C+1H 退化成 2C+1H，因为 rewriter 在修其它 axis 时引入了新 critical。

**你的做法**：A1 状态机（见 `regression_guard.py`）：

```python
if curr.critical_plus_high > prev.critical_plus_high:  # 回归
    if retries_used < max_regression_retries (=2):
        revert_to(spec_snapshots[last_good_iteration])
        rewrite_with_regression_feedback(...)
    else:
        halt + needs_review
elif curr.critical_plus_high < prev.critical_plus_high:  # 改善
    update last_good_iteration
elif stagnant:
    if 连续 3 轮: halt + needs_review
```

关键设计点：
1. **回归不烧 no-progress 预算**——回归是另一个失败模式，独立计数
2. **last_good 只在 improved 时更新**——确保 revert 永远回到"严格更好"的状态
3. **needs_review 不是 fail**——spec 仍然落盘，trace 完整，只是告诉下游人工 review

**别的做法为什么不行**：
* "每次都重写"——发散
* "回归就停"——过于保守，第一次回归就放弃
* **revert + 限定预算 + 回归感知 prompt** 是经验上最稳的折中。

## 8. 面试常见问题 Q&A（提前演练）

> **温馨提醒**：面试问得越深越好——下面这些都是真实可能被问到的"追问"，准备好你的弹药。

### Q1: 19 个防御机制听起来很多，是不是过度工程？

**A**: 不是。每一个防御都对应**至少一个我在 6 个 Mealie 真实 case 上亲眼见过的失败模式**——见 `DEFENSES.md` 每条的 "Designed to catch" 字段。没有任何一个防御是"想到了就加"。
更关键的是：4 个机械校验（A4/A5/B1/B3）的"失败模式"在装上后是**结构上不可能再出现**的——不是降低概率，是物理上消失。换句话说，这是**用 200 行 Python 一次性彻底消灭一类 bug**，而不是反复打补丁。从这个角度看，反而是**最经济的工程投入**。

### Q2: 为什么不直接用一个大模型 + 长 prompt 解决？

**A**: 试过，不行。原因：
1. **同模型自评偏见**——同家族模型自评分数虚高 5–18%（论文证据），单模型方案无法 cross-check
2. **prompt 提示对硬约束的修正率只有 60–75%**——LLM 经常忽略禁令；改成 pydantic 抛错后修正率 ≈ 100%
3. **单模型无法做 5 视角并行探索**——拆成 5 个 explorer 各自带工具调用，单次输出能力 5×，且各自有独立的 token budget
4. **单模型无法识别"收敛底线"**——LLM 倾向于"再试一次"，永远不会主动叫停

### Q3: 这跟 LangChain / spec-kit / Cursor 有什么本质差别？

**A**: 关键差别在**评审与防御**这一层。
* LangChain 是 framework，提供 agent / chain 抽象，**不提供质量保证**——你写啥它跑啥
* spec-kit / Cursor 的 spec 生成是"单次 LLM + 模板"，没有跨家族评审、没有机械校验、没有迭代收敛检测、没有红队
* 本系统的 9 阶段、4+1 reviewer、19 项防御不是"功能多"——是为了让 **spec 的失败模式可观测、可量化、可阻断**

任何一个 leak 的 critical/high 都可以被定位到具体的 stage + reviewer + validator。这是 research-grade 而非 chatbot-grade 的设计取向。

### Q4: 测试覆盖率 79.84% 是不是不够？为什么不到 100%？

**A**: 实测 `coverage report` = `TOTAL 4629 stmts, 933 missed, 79.84%`。
未覆盖 ~20% 集中在两类：
1. **LLM provider 真 API 调用路径**（`anthropic_provider.py` / `openai_provider.py` 的网络层）——必须接真 key 才能跑，CI 不便接，这是**有意识的决定**
2. **`tools/references.py` 的 tree-sitter 失败兜底分支**——需要恶意构造的语法错误源文件触发

**所有核心业务逻辑（schemas / validators / orchestrator / reviewers / regression_guard）都 ≥ 90% 覆盖**。79.84% 是经过审视的"该测的全测了"的数字，不是"差点 100%"。这也是**诚实标注**，没有刷数字。

### Q5: case5 没收敛是不是 bug？

**A**: 不是。case5 是 cross-domain 多技术栈混合（scheduler + multitenant + i18n + event bus）；5 轮后剩下的 5 个 critical+high 是 **transactional outbox vs internal commits 的设计权衡** 和 **"当天无 meal plan 时 target list 走哪条"的产品语义** —— 这些**没有 LLM 能在不询问产品经理的情况下回答**。

系统识别出"剩下的问题都需要 human/PM input"并 halt 是**正确行为**。详见 `specs/case5-live-iter1-20260619T175133Z/FINDINGS.md` 第 60-82 行 Finding 6 + Finding 7。这就是"收敛底线"概念存在的意义——好的系统知道什么时候**该停下问人**，而不是继续幻觉式地"修改"。

### Q6: 跨家族评审增加了多少成本？

**A**: 单次 v1→v2 完整迭代，所有防御开启时的 LLM round-trip 计数 ≈ **15–24 次**（writer 1 + 5 explorers + 可能 3 个 B2 targeted re-explore + 4–5 reviewers + B4 meta + rewriter 1–5）。
按 Claude Opus 4.7 + GPT-5.5 估价，**单 case 约 $0.5–1.5**（视 spec 长度）。
比裸 LLM 单次贵 ~15×，换来 **4 类 leakage 的结构性消除**——见 Q1。在 AI 自动开发流水线里，**spec 阶段一旦 leak，下游 plan/code/test agent 的浪费是指数放大的**，所以这 15× 成本完全划算。
如果只是 prototype 阶段，可加 `--single-explorer --single-candidate --single-reviewer` 进入 MVP 模式（约 3× 快 5× 便宜）。

### Q7: 如何保证写作和评审的 prompt 没有"互相串通"？

**A**: 三道防护：
1. **跨家族**——writer 和 reviewer 是不同公司模型，训练分布不同
2. **Prompt 物理分离**——`prompts/` 下每个 agent 一个独立 .md 文件，git 版本化，**没有任何 prompt 引用其它 agent 的输出格式**
3. **Schema 隔离**——writer 输出 `Spec`，reviewer 输出 `Review`，两者由 pydantic 隔离，writer 无法"暗示"reviewer 怎么评

### Q8: 如果 LLM 输出的 JSON 不合规怎么办？

**A**: `devloop/llm/json_helpers.py::call_strict_json` 实现：
1. 第一次输出后用 pydantic 解析
2. 失败则把 `ValidationError` 字符串喂回去当 prompt 让 LLM 修
3. 最多 N 次（默认 3）
4. 仍失败则 `raise SubAgentFailedError`——**不静默 skip**

为什么不静默？因为早期版本静默 skip 让一个 reviewer 失败时其它 reviewer 仍然通过，最终 spec 漏掉 1 个 critical。改成 `raise` 后**要么所有 reviewer 全部成功，要么整个 run 抛错退出**。没有"半失败但 spec 看起来正常"的中间态。这是 **fail loud, never silent skip** 哲学的具体体现。

### Q9: 你这个项目有没有什么暴露的安全问题？

**A**: 有，且已**公开记录**在 `specs/CAPABILITY_BOUNDARY_REPORT.md:586-590`：
1. **A5 path-traversal**——`code_references.file = "../../../etc/passwd"` 当前能通过校验（已有测试文档化，未修），外发客户前必须堵
2. **真 API key 跑通**——目前 6 个 case 是 Copilot CLI sub-agent 跑的，不是直接调真 Anthropic/OpenAI key（机械防御 100% 不受影响，但 LLM-dependent 路径需复核）
3. **A3 加强**——`BlockingDecision` 当前是 schema scaffolding，不能强制 writer 使用

**这种"公开未修缺陷"的态度本身就是工程严谨性的体现**——不藏，不修饰，给用户决策依据。

### Q10: 项目对你个人最大的提升是什么？

**A**（这道题没标准答案，给一个范例）：
1. **从"让 AI 做事"到"约束 AI 做事"的范式转变**——以前总想着提示工程让 LLM 更准，做了这个项目后明白：**真正可靠的系统是把不变式从 prompt 升级到代码**
2. **跨学科融合**——把 LLM-as-judge bias 论文（学术）+ pydantic schema（工程）+ tree-sitter（编译原理）+ 状态机（控制论）拼成一个系统
3. **"失败优先"思维**——19 个防御里每一个都是从"一次见到的失败"反推出来的，而不是从"听起来该有"想出来的。这让我习惯了**先看真实数据，再做设计决策**
4. **学会承认系统的边界**——case5 的"收敛底线"概念让我理解：**好的系统知道何时该问人**，比假装通过更诚实

### Q11: 假如让你从头再做一遍，你会改什么？

**A**（展示反思能力）：
1. **A1 改成 per-axis 监控**——目前只看总 critical+high，case5 v1→v2 的 completeness 退化被忽略
2. **D3 分段重写器接 A/B 实验**——目前 opt-in 默认关闭，但实测可能在大 spec 上有 30% 收益
3. **A5 path-safety 立即修**——这是公开已知 bug
4. **ScopeType literal 扩 6 个值**——case5 撞到了 enum 不够用
5. **C3 perspective auto-select 加 LLM 推荐**——目前是规则匹配，可能漏掉新场景

诚实承认这些点，比硬辩"完美"加分得多。

### Q12: 这个项目可以扩展到哪些方向？

**A**: 三个方向：
1. **接 plan / code / test 阶段**——本仓库只做 spec phase，下游还有大半条流水线
2. **跨语言支持**——目前 tree-sitter 覆盖 Python/JS/TS/Go/Rust/Java；可加 C#/C++/Kotlin
3. **接入企业内部 LLM**——`ModelRouter` 的 provider 抽象已经写好，加一个 `internal_provider.py` 即可
